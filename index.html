Deep Agent + 임베딩
REALTIME
사용자 자연어 입력 (Frontend)
"3시에 홍대, 예산 10만원, 카페→식사→술, 분위기 좋은 곳으로 저녁 8시까지"
↓
Deep Agent (deepagents 라이브러리)
① 사용자 입력 분석 (LLM)
시간: 15:00 ~ 20:00 지역: 홍대 예산: 100,000원 순서: 카페 → 식사 → 술 분위기: 감성적
② 임베딩 변환 (deepagents 자동)
모델: ko-sroberta-multitask (한국어 특화) "분위기 좋은 카페 홍대" → [0.2, 0.8, 0.1, 0.9 ...]
③ Tool 트리거 판단
날씨 API → 오늘 날씨 확인 동선 API (카카오맵) → 이동 시간 계산 개인화 필터 → 커플 취향 교집합
핵심: 자연어를 그대로 수신 → LLM이 의도 파악 → 임베딩 변환까지 deepagents가 자동 처리. 백엔드 파싱 불필요.

벡터 DB (Qdrant)
PRE-PROCESS — 주기적 업데이트
벡터 DB 저장 구조 (FAISS / Qdrant)
Point = 벡터 + 페이로드 vector: [0.2, 0.8, 0.1, ...] payload: name: "카페 온도" category: "카페" location: "홍대" price: 10000 hours: "10:00~22:00" is_open: true review: "감성적인 조용한 카페"
+
SQL 사용자 정보 (정형 데이터)
users 테이블 ────────────────────────── user_id: 42 name: "망키손" age: 24 couple_prefs 테이블 ────────────────────────── user_id: 42 partner_id: 43 mood: "감성적" food_type: "한식, 이탈리안" avoid: "시끄러운 곳" budget: 100000 age_group: "20대초반"
핵심: 벡터 DB (장소 후보) + SQL (커플 취향) → 두 데이터를 합쳐서 프롬프트 Context로 LLM에 전달.
REALTIME — 사용자 요청 시
유사도 검색
코사인 유사도 계산
입력 벡터 [0.2, 0.7, 0.1, 0.8 ...] ↓ 저장된 모든 벡터와 비교 ↓ 유사도 높은 순 정렬
후보 장소 반환
1위: 카페 온도 (0.94) 2위: 카페 을지로 (0.91) 3위: 감성 커피숍 (0.88) ... → 상위 N개 페이로드 반환
핵심: 카테고리 없이 의미 기반 자동 군집. 유사도가 높을수록 문맥적으로 가까운 장소. 벡터 = 검색용, 페이로드 = LLM에 전달할 실제 데이터.

미결 질문 — 팀과 확정 필요
Q1. 벡터 DB 뭐 쓸까?
FAISS (로컬)
✓ 설치 간단, 빠름 ✓ 외부 서버 불필요 ✗ 필터링 기능 약함 ✗ 실시간 업데이트 불편 → 프로토타입·중간발표용
vs
Qdrant (Docker)
✓ 페이로드 필터 강력 ✓ REST API 지원 ✓ 실시간 업데이트 가능 ✗ Docker 세팅 필요 → 최종 배포용
현재 생각: 중간발표까진 FAISS로 빠르게 → 이후 Qdrant 마이그레이션. 팀 합의 필요.
Q2. SQL은 뭐 써야 하지?
SQLite
✓ 설치 불필요, 파일 기반 ✓ 로컬 개발 초간단 ✗ 동시 접속 약함 → 개인 테스트용
vs
PostgreSQL
✓ 실제 서비스 표준 ✓ 동시 접속 강함 ✓ 백엔드 팀 익숙할 가능성↑ ✗ 서버 세팅 필요 → 팀 프로젝트·배포용
vs
MySQL
✓ 국내 많이 사용 ✓ 백엔드 팀 친숙 ✗ PostgreSQL보다 기능 제한적 → 팀이 익숙하면 OK
현재 생각: 백엔드 팀이 익숙한 걸로 결정하되, PostgreSQL 권장. AI 입장에선 조회 API만 호출하면 뭐든 상관 없음.
Q3. AI 아웃풋 → 프론트 연결 방식
Option A — AI가 직접 API 노출
Frontend ↓ POST /recommend FastAPI (AI) ↓ JSON 응답 Frontend
장점
✓ 구조 단순 ✓ 백엔드 팀 의존 없음 ✓ AI 파트 독립 개발 가능 ✓ 속도 빠름
단점
✗ 인증·보안 직접 처리 ✗ 백엔드 기능 필요 시 연결 복잡해질 수 있음
vs
Option B — 백엔드 경유
Frontend ↓ POST /recommend Backend (Spring/Django) ↓ 내부 호출 FastAPI (AI) ↓ JSON 응답 Backend ↓ Frontend
장점
✓ 인증·로깅 백엔드가 담당 ✓ 팀 역할 분리 명확 ✓ 확장 시 유연
단점
✗ 레이어 하나 더 ✗ 백엔드 팀과 연동 필요 ✗ 응답 속도 약간 느림
현재 생각: 중간발표까진 Option A (직접 연결)로 빠르게. 최종 배포 때 Option B 고려.
Q4. 사용자 취향 — 어떻게 받을까?
Option A — 온보딩 선택 (권장)
회원가입 시 취향 선택 ────────────────── 분위기: [ ] 감성 [ ] 활기찬 음식: [ ] 한식 [ ] 양식 예산: [ ] ~5만 [ ] ~10만 연애단계: [ ] 초반 [ ] 1년+ ↓ Backend가 SQL에 저장 Agent는 읽기만 함
✓ 구현 단순, 데이터 명확
vs
Option B — AI 분석
대화형으로 취향 파악 ────────────────── "어떤 분위기 좋아해요?" → 자유 입력 → Agent가 분석 후 SQL에 자동 저장 ↓ 복잡도 높음
✗ 구현 복잡, 오차 가능성
현재 생각: Option A로 확정. 저장은 백엔드, Agent는 조회만. 프론트한테 온보딩 화면 요청 필요.
